## Upgrade Notes

{{< notice warning >}}
v1.24.0 is a **security-hardening release**. Several primitives that were previously accepted are now rejected at admission, so configurations that relied on them will break on upgrade. Audit before upgrading:

- **Cross-namespace references are rejected.** A `Function`, `Package`, or `KubernetesWatchTrigger` that points `spec.environment.namespace`, `spec.package.packageref.namespace`, or `spec.namespace` at a *different* namespace is now refused. Empty (same-namespace) values are unaffected.
- **Dangerous PodSpec fields are rejected** on `Environment` and `Function` (`hostNetwork`, `hostPID`, `hostIPC`, hostPath volumes, container `privileged`/`allowPrivilegeEscalation`, dangerous Linux capabilities, `serviceAccountName` override), and on `MessageQueueTrigger` (PodSpec is now allowlisted to `nodeSelector`, `tolerations`, `affinity`, `runtimeClassName`, and per-container `resources`).
- **`KubernetesWatchTrigger` with an empty `spec.namespace` now watches its own namespace**, not all namespaces.
- **Cross-origin browser requests are denied by default** across all Fission HTTP listeners. SPAs that legitimately call a trigger cross-origin must opt in via the new `HTTPTrigger.spec.corsConfig`.

See the [Upgrade Guide](/docs/installation/upgrade/#upgrade-to-124x-release) for the full procedure.
{{< /notice >}}

## Deprecations/Removals

- We continue to support Kubernetes 1.28 and above (`kubeVersion: ">=1.28.0-0"` in the Helm chart). No change in supported Kubernetes versions from v1.23.0.
- Cross-namespace `Environment`/`Package` references and dangerous PodSpec fields on `Environment`, `Function`, and `MessageQueueTrigger` are no longer accepted â€” see Upgrade Notes above.
- The **fission-builder ServiceAccount token** is no longer mounted into user builder containers (mirrors the fission-fetcher change in v1.23.0).
- License headers across the codebase were converted to SPDX identifiers; the one-time strip script has been removed.

## Highlights

- **Confused-deputy hardening across the CRD surface.** The admission webhooks (with controller-side guards as defence in depth) now reject cross-namespace `Environment`/`Package` references on `Function`, `Package`, and `KubernetesWatchTrigger`, closing several high/critical advisories where a low-privilege subject could target another namespace.
- **PodSpec injection is closed.** `Environment` and `Function` reject a denylist of node-escape primitives (`hostNetwork`/`hostPID`/`hostIPC`, hostPath volumes, `privileged`, dangerous capabilities, SA override), `MessageQueueTrigger` PodSpec is reduced to a small allowlist, and `Environment` runtime/builder container `securityContext` is validated. Benign customization (`image`, `command`, `env`, `resources`, `nodeSelector`, â€¦) is unaffected.
- **Cross-origin defense.** A shared middleware adds `X-Content-Type-Options: nosniff` and `Vary: Origin` to every response and denies cross-origin browser preflights on internal listeners. A new opt-in `HTTPTrigger.spec.corsConfig` lets SPAs allowlist origins per trigger without baking CORS into the function body.
- **CRD status conditions.** Every Fission CRD now carries a standard `metav1.Condition` array and `observedGeneration`, with Server-Side Apply list markers on the fields where atomic merging was wrong. This is the schema groundwork for richer status reporting.
- **fission CLI gains condition-aware UX:** status conditions surface in `list`/`get`, `-o json|yaml|wide` output formats, `wait --for=condition=...` for condition-bearing resources, and `spec apply --dry-run`.

## Fixes

- `MessageQueueTrigger` no longer materializes referenced Secret values into the connector pod template â€” env vars now resolve via `secretKeyRef` at pod start using the connector's own ServiceAccount.
- Empty `KubernetesWatchTrigger.spec.namespace` is coerced to the trigger's own namespace instead of watching cluster-wide.
- Routine security sweeps refreshed Go dependencies (including go-git `5.19.1`) and bumped workflow tool versions (skaffold `v2.21.0`, goreleaser-action `v7.2.2`).
- Helm chart published as **1.24.0**, versioned independently from the app version.
